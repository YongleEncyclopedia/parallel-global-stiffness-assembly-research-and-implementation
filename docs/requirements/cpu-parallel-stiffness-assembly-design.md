# CPU 平台并行整体刚度矩阵组装可复现实验平台需求文档

## 1. 文档信息

- 文档日期：2026-04-22
- 文档状态：需求定义 v2
- 适用对象：项目执行者、合作导师、后续代码维护者
- 文档目标：明确 CPU 平台并行组装整体刚度矩阵项目当前阶段的研究范围、技术路线、功能需求、实验设计、验收标准与风险控制

## 1.1 当前阶段定位

本项目当前已经不是“从零做 CPU 并行组装原型”的阶段，而是进入了**把现有 CPU 主线代码推进成可复现实验平台**的阶段。

因此，本阶段工作的重点是：

- 保持唯一主线目录不再分叉
- 补齐真实工程网格上的实验矩阵
- 输出更完整的 benchmark 指标和实验图表
- 让算法说明、README、需求文档和实验摘要彼此一致
- 让后续开发者或大模型可以直接在仓库中复现实验

## 2. 项目背景

### 2.1 研究背景

根据 2026 年 1 月月度汇报中的既有工作，本课题前一阶段重点在 GPU 平台，已经完成了多类 GPU 并行组装策略的调研与实现；同时也梳理了 CPU 侧的典型思路，包括：

- 图着色法（Graph Coloring）
- 直接累加法（Addto / Atomics）
- 区域分解法（Domain Decomposition）

当前任务目标已经明确转向：不再以 GPU 为主要研究对象，而是面向多核 CPU 平台系统调研并验证多类并行组装算法的可行性，最终形成一套可在真实工程网格上复现实验结论的 CPU 并行组装研究平台。

### 2.2 现有代码与资料基础

当前工作区内与本项目直接相关的基础如下：

1. `fem_assembly`
   - 已实现二维规则四边形网格下的图着色 + OpenMP 并行组装原型。
   - 已具备串行/并行结果比对、性能测试、CSV 导出和图着色分析。
   - 当前更接近“算法原型验证代码”，而非真实工程网格研究平台。

2. `parallel_global_stiffness_assembly/cpu_parallel_stiffness_assembly`
   - 已建立统一的 benchmark / assembler factory 主框架。
   - 已支持 3D `Hex8` / `Tet4`、每节点 3 个自由度、CSR 结构预计算、scatter plan、`.inp` 解析、CPU 串行基线和多类 CPU 并行算法。
   - 当前已经是唯一主线目录，不再只是“代码骨架”。

3. `3d-WindTurbineHub.inp`
   - 为真实工程网格输入文件。
   - 初步统计结果：约 `228385` 个节点，约 `1113684` 个 `C3D4` 单元。
   - 这意味着项目后期必须处理百万级单元、非结构化四面体网格和 Abaqus `.inp` 输入格式。

### 2.3 当前问题

虽然当前主线已经具备 CPU 并行算法、`.inp` 输入和 benchmark 基础能力，但仍存在以下缺口：

- 真实工程网格上的算法覆盖还不完整；
- benchmark 输出指标仍不够丰富；
- 图表和实验摘要还不够成熟；
- 项目文档还没有完全切换到“可复现实验平台”叙事；
- 一部分 GPU 历史内容虽然保留，但需要明确隔离为 legacy 参考。

## 3. 项目目标

### 3.1 总体目标

将当前主线代码强化为一套面向多核 CPU 平台的并行整体刚度矩阵组装可复现实验平台，系统比较多类共享内存并行组装算法在不同网格规模和不同输入类型下的正确性、性能、扩展性、内存开销与工程可用性。

### 3.2 分目标

1. 建立统一的 CPU 并行组装实验框架，保证不同算法在相同网格、相同矩阵结构和相同测试流程下公平对比。
2. 在简单三维块体网格和真实工程网格上完成多算法统一 benchmark。
3. 输出结构化实验结果、图表与摘要，回答“哪类算法在 CPU 上更可行、代价是什么、适合什么规模和场景”。
4. 保持对 `macOS + Mac Studio` 的本机可重复验证，并为后续 `Windows + Intel` 迁移保留一致接口。

### 3.3 平台目标

项目的执行顺序必须明确采用“两阶段平台策略”：

1. 第一阶段在本机 `macOS + Mac Studio` 环境完成原型验证与文档驱动开发。
2. 第二阶段为后续迁移到 `Windows + Intel U7 265KF` 平台预留兼容性和构建路径。

因此，本项目从一开始就不是单平台实验脚本，而是需要同时考虑 `macOS Apple Silicon` 与 `Windows Intel x86_64` 适配边界的 CPU 研究项目。

### 3.4 项目最终要回答的问题

1. 在多核 CPU 上，图着色是否仍然是最合适的基线方案？
2. 直接累加类方法是否能利用 CPU 缓存与硬件同步机制取得更好表现？
3. 线程私有缓冲或子域划分方法是否能在真实大网格上优于图着色？
4. 当网格从规则块体切换到真实 `C3D4` 工程网格后，各算法的趋势是否保持一致？
5. 工程上应该优先保留哪一类算法作为后续深入优化的主线？

## 4. 范围界定

### 4.1 本项目范围内

- 共享内存多核 CPU 平台上的并行整体刚度矩阵组装研究
- `macOS (Apple Silicon)` 与 `Windows (Intel x86_64)` 的跨平台适配约束设计
- 统一 benchmark 框架搭建
- 简单结构化三维测试网格生成
- Abaqus `.inp` 中 `*NODE` 与 `*ELEMENT, TYPE=C3D4` 的解析支持
- `Tet4` 为主、`Hex8` 为辅的测试能力
- 串行基线、图着色、直接累加、局部缓冲/归并、子域分解类算法对比
- 正确性、速度、扩展性、预处理代价、内存代价的系统评估

### 4.2 本项目暂不纳入

- 分布式内存并行（MPI）
- GPU 新算法继续扩展
- 全套商业求解器级功能
- 所有 Abaqus 关键字和所有单元类型的完整支持
- 求解器阶段（线性方程组求解）的系统优化
- 高阶单元、非线性材料、接触等复杂物理模型

## 5. 方案比较与推荐路线

## 5.1 方案 A：直接在 `fem_assembly` 上继续扩展

优点：

- 已有 CPU 图着色实现
- 代码较小，理解成本低
- 适合快速验证图着色思路

缺点：

- 目前是二维规则四边形原型，和真实目标差距大
- 缺少统一的多算法可插拔框架
- 缺少与 GPU 阶段统一的 benchmark 组织方式
- 迁移到 `Tet4`、`.inp`、真实工程算例时改造量大

结论：

- 适合作为“图着色 CPU 基线实现来源”，不适合作为主平台继续承载新阶段全部工作。

## 5.2 方案 B：基于 `cpu_parallel_stiffness_assembly` 扩展 CPU 后端

优点：

- 已有统一的 `IAssembler` / `AssemblerFactory` / benchmark 主程序
- 已具备 3D `Tet4` / `Hex8`、3 自由度节点、CSR 框架和 CPU 串行基线
- 结构更利于统一比较 CPU 与 GPU 路线
- 更接近真实工程网格研究平台

缺点：

- 当前 CPU 侧只有串行实现，需要补全并行后端
- 目前 `load_from_file()` 不是 Abaqus `.inp` 解析器
- 当前符号组装与局部索引查找实现仍偏原型，需增强大规模可用性

结论：

- 这是推荐主路线。

## 5.3 方案 C：新建独立 CPU 项目

优点：

- 架构可完全按 CPU 目标重设计
- 历史包袱最少

缺点：

- 重复造轮子
- 会与现有 GPU 框架割裂
- 不利于后续统一对比与维护

结论：

- 不推荐，除非现有框架证明无法承载需求。

## 5.4 推荐决策

推荐采用方案 B：

- 以 `parallel_global_stiffness_assembly/cpu_parallel_stiffness_assembly` 作为主平台；
- 将 `fem_assembly` 中已验证的图着色思想迁移为一个 CPU 后端；
- 在统一框架中新增其它 CPU 并行算法；
- 用同一套 benchmark 流程完成简单网格与工程网格的分阶段测试。

## 6. 总体技术要求

### 6.1 总体架构要求

系统应至少划分为以下层次：

1. 输入与网格层
   - 规则块体网格生成
   - Abaqus `.inp` 解析
   - 节点/单元/自由度映射管理

2. 矩阵结构与预处理层
   - CSR 结构预计算
   - 可选的单元散射地址预计算
   - 图着色或分区等算法预处理

3. 数值组装算法层
   - 串行基线
   - 图着色并行
   - 直接累加类并行
   - 局部缓冲归并类并行
   - 子域/owner-computes 类并行

4. benchmark 与结果输出层
   - 参数解析
   - 多轮测试
   - 正确性验证
   - CSV/JSON 输出
   - 图表和结论生成

5. 平台适配层
   - 编译器与 OpenMP 能力探测
   - macOS / Windows 路径与脚本差异隔离
   - 平台相关宏与可选特性开关

### 6.2 核心设计原则

1. 算法可插拔：新增一种算法不应重写 benchmark 主流程。
2. 阶段可拆分：必须分别统计符号组装、算法预处理、数值组装、总时间。
3. 结果可复现：相同输入、相同线程数、相同编译配置下应得到稳定可比较结果。
4. 对比公平：不同算法应共享相同网格、相同 CSR 结构和相同局部刚度模型。
5. 先简后难：先规则块体，再真实 `.inp` 工程网格。
6. 平台隔离清晰：平台相关逻辑必须集中管理，避免源码中散落大量临时条件分支。
7. 默认跨平台：任何新增脚本、构建说明和路径约定都应优先考虑同时兼容 macOS 与 Windows。

## 7. 详细功能需求

### 7.1 输入与网格需求

#### FR-001 结构化测试网格生成

系统必须支持生成三维块体规则网格，并至少支持以下模式：

- `Hex8`
- `Tet4`（主模式，优先级高于 `Hex8`）

其目的是快速形成可控、可扩展、可重复的早期实验数据集。

#### FR-002 工程网格输入

系统必须支持读取 Abaqus `.inp` 文件中的以下内容：

- `*NODE`
- `*ELEMENT, TYPE=C3D4`

系统可以忽略但必须安全跳过以下类型内容：

- 注释行 `**`
- `*NSET`、`*ELSET`
- 材料、载荷、边界条件等与“组装算法验证”无关的段落

#### FR-003 索引与数据转换

系统必须将 Abaqus 的 1-based 编号转换为内部 0-based 编号，并保证：

- 节点映射正确
- 单元连接关系正确
- 不因跳号、空洞编号或多余 section 导致解析错误

#### FR-004 网格统计输出

对于任意输入网格，系统必须输出至少以下统计信息：

- 节点数
- 单元数
- 单元类型
- 自由度总数
- CSR 非零元数
- 预估内存消耗

### 7.1A 跨平台输入输出与路径需求

#### FR-004A 路径兼容性

系统中的输入文件路径、结果输出路径和配置文件引用必须兼容：

- macOS 路径分隔符与 shell 环境
- Windows 路径分隔符与命令行环境

代码中不得把平台相关路径写死为某一系统专用格式。

#### FR-004B 大文件策略

真实工程案例如 `3d-WindTurbineHub.inp` 这类大体积输入文件应优先通过 Git LFS 以仓库内固定路径提供，例如：

- `examples/3d-WindTurbineHub.inp`

系统必须提供：

- 清晰的 Git LFS 拉取说明
- 缺失输入文件或仍为 LFS pointer 时的错误提示
- 用于本地验证的小型替代样例或格式说明

### 7.2 矩阵结构与预处理需求

#### FR-005 统一 CSR 结构

所有算法必须共享统一的全局矩阵存储格式。第一阶段统一采用 CSR，以确保：

- 与现有框架兼容
- 与求解器常用格式一致
- 算法对比具有统一基础

#### FR-006 符号组装与数值组装分离

系统必须将：

- CSR 结构预计算（symbolic assembly）
- 数值填充（numeric assembly）

分开统计和控制，避免将两者混在一起导致性能结论失真。

#### FR-007 局部到全局地址映射预计算

系统应支持预计算“单元局部矩阵条目 -> CSR 中目标位置”的散射地址映射，用于降低数值组装阶段反复二分查找 `find_col_index()` 带来的干扰。

这样做的目的不是取消原始实现，而是让实验能够区分：

- 冲突消解开销
- 地址查找开销
- 单元刚度计算开销

#### FR-008 大规模预处理可用性

对于百万级 `Tet4` 网格，系统不得只依赖当前原型式的重型动态容器实现作为唯一方案；至少应满足以下之一：

- 提供更轻量的符号组装/邻接构建实现；
- 提供大规模模式下的替代预处理策略；
- 明确区分“教学原型模式”和“工程规模模式”。

### 7.3 局部刚度计算需求

#### FR-009 双模式局部刚度计算

系统必须支持两类局部刚度模式：

1. `simplified_kernel`
   - 用于隔离“组装策略”本身的差异
   - 允许继续使用当前简化的 `Tet4/Hex8` 局部刚度模型

2. `physics_like_tet4_kernel`
   - 用于在真实 `C3D4` 工程网格上进行更可信的 CPU 可行性验证
   - 至少应保证对单元几何有响应，并与线性四面体刚度矩阵计算逻辑一致

要求 benchmark 结果中明确记录当前使用的 kernel 模式。

### 7.4 算法需求

#### FR-010 串行基线算法

系统必须保留 CPU 串行组装作为唯一正确性基准和加速比基准。

#### FR-011 图着色并行算法

系统必须实现图着色并行组装算法，作为 CPU 侧历史基线，并满足：

- 支持从 `fem_assembly` 迁移已有逻辑
- 颜色构建时间单独统计
- 支持颜色数、各颜色规模、负载分布输出
- 结果应具备确定性或可解释的稳定性

#### FR-012 直接累加类算法

系统必须实现一类直接累加并行算法，优先级建议如下：

1. 基于 `OpenMP atomic` 的直接累加
2. 如原子性能不足，可增加按行锁或细粒度锁变体

该算法的目标是验证在 CPU 上保留自然访问顺序、减少复杂预处理后，是否能超过图着色基线。

#### FR-013 线程私有缓冲 + 归并算法

系统必须实现一类“线程私有写入、最后归并”的算法。可选实现方式包括：

- 线程私有 COO 缓冲 + 排序归并
- 线程私有局部 CSR/哈希表 + 合并
- 分块归并

该类算法的研究重点是：

- 减少并发写冲突
- 以更高内存占用换取更少同步
- 评估归并开销是否值得

#### FR-014 子域/owner-computes 类算法

系统应实现一类基于网格划分或 owner-computes 思想的 CPU 并行算法，基本要求包括：

- 将单元划分给若干线程负责
- 内部自由度无冲突优先本地处理
- 边界贡献通过归并、归约或同步处理

该算法可作为“较高复杂度但更工程化”的候选路线。若时间不足，可列为第二优先级，但必须在文档中保留设计接口与实验预留位。

#### FR-015 算法扩展接口

算法枚举、配置参数和 benchmark 框架必须支持后续继续新增 CPU 算法，而不需要重写主程序。

### 7.5A 跨平台构建与运行需求

#### FR-018A 构建系统统一

项目必须优先采用跨平台构建系统，推荐以 `CMake` 作为统一入口，避免将核心工作流绑定在单一平台脚本上。

#### FR-018B 编译器支持矩阵

需求文档与后续实现必须至少面向以下编译环境设计：

- `macOS + Clang/AppleClang`
- `Windows + MSVC`

如引入 `OpenMP`，必须在需求和实现中明确说明：

- macOS 上的可用编译方式与依赖前提
- Windows/MSVC 上的启用方式
- 当 `OpenMP` 不可用时的降级或提示策略

#### FR-018C 脚本双通道

若项目需要自动化脚本，至少应满足以下之一：

- 提供跨平台 Python 脚本作为主入口；
- 或同时提供 macOS/Linux shell 与 Windows `bat`/`PowerShell` 入口；

不得把核心实验流程仅绑定到单一平台专属脚本。

#### FR-018D 平台信息输出

benchmark 结果必须记录运行平台信息，至少包括：

- 操作系统
- CPU 架构
- 编译器
- 线程库/并行后端配置

这样才能正确解释 Apple Silicon 与 Intel x86_64 平台上的性能差异。

### 7.5 benchmark 与结果输出需求

#### FR-016 测试参数化

系统必须支持通过命令行或配置文件指定以下参数：

- 网格来源（生成 / `.inp`）
- 网格规模
- 单元类型
- kernel 模式
- 算法类型
- 线程数
- 预热次数
- 正式测试次数

#### FR-017 结果输出

每次 benchmark 至少输出以下字段：

- 算法名
- 网格名/算例名
- 单元类型
- 节点数、单元数、自由度数、NNZ
- 线程数
- 符号组装时间
- 预处理时间
- 数值组装时间
- 总时间
- 相对误差
- 加速比
- 内存占用
- 运行是否成功

输出格式至少包含：

- CSV
- 控制台摘要

如有余力，可增加 JSON 便于后处理。

#### FR-018 结果可视化

系统应支持自动或半自动生成以下图表：

- 线程数-时间曲线
- 线程数-加速比曲线
- 线程数-并行效率曲线
- 不同算法在同一算例上的对比图
- 预处理时间与数值组装时间构成图

### 7.6 正确性验证需求

#### FR-019 基础正确性

所有并行算法都必须与串行基线进行数值比对，并至少报告：

- Frobenius 相对误差
- 最大绝对误差
- 结构一致性检查

#### FR-020 非确定性算法判定标准

对于直接累加或归并类算法，允许由于浮点累加顺序差异导致的微小非位级一致，但必须满足：

- 相对误差不超过预设阈值
- 多次运行误差水平稳定
- 不允许出现结构错误、明显发散或线程数变化导致的大幅异常

## 8. 非功能需求

### 8.1 性能需求

本项目的性能目标分为“可行性底线”和“竞争性目标”两级。

#### 可行性底线

1. 在中大规模规则 `Tet4` 网格上，至少一种并行算法相对串行基线达到稳定加速。
2. 在 `3d-WindTurbineHub.inp` 上，至少两种并行算法能够完整运行并通过正确性验证。
3. 在 `3d-WindTurbineHub.inp` 上，至少一种并行算法的数值组装时间相对串行基线达到 `1.5x` 以上加速。

#### 竞争性目标

1. 在大规模规则 `Tet4` 网格上，至少一种非图着色算法超过图着色基线。
2. 在 `3d-WindTurbineHub.inp` 上，至少一种算法相对图着色基线的数值组装时间提升达到 `20%` 以上。
3. 预处理时间不能大到抵消数值组装阶段的大部分收益，因此必须同时报告“数值组装加速”和“端到端加速”。

### 8.1A 跨平台性能解释要求

项目必须明确区分以下两类结论：

- 算法层结论：某算法在相同平台内相对于其它算法的表现；
- 平台层结论：同一算法在 Apple Silicon 与 Intel x86_64 上的表现差异。

需求文档和后续报告不得把“平台差异”误写成“算法优劣”。

### 8.2 可扩展性需求

系统必须支持多线程扩展性测试，至少覆盖：

- `1`
- `2`
- `4`
- `8`
- `16`

若开发机物理核心数更高，应继续扩展至物理核心上限。结果中必须区分：

- 物理核心扩展
- 超线程扩展（若测试）

### 8.3 内存需求

系统必须控制中间态内存开销，避免在真实工程案例上出现以下问题：

- 因中间 COO 过大导致无法运行
- 因预处理数据结构过重导致主存压力过大
- 因频繁动态分配导致显著性能畸变

对于每类算法，必须报告或估算其额外内存占用。

### 8.4 可维护性需求

代码必须遵循以下要求：

- 输入解析、CSR 构建、算法实现、benchmark、可视化分层清晰
- 算法新增时对现有代码侵入最小
- 关键实验配置能通过固定参数复现实验

### 8.5 可移植性需求

系统必须控制以下跨平台风险：

- 编译器特性差异导致的不可编译
- OpenMP 支持差异导致的不可运行
- 路径、编码、换行符差异导致的脚本失效
- ARM64 与 x86_64 在对齐、向量化和性能特征上的差异被忽略

为此，后续实现必须优先选择标准 C++17、跨平台构建工具和平台无关的数据路径约定。

## 9. 实验设计要求

### 9.1 分阶段实验集

#### 第一阶段：快速功能验证

目标：

- 跑通流程
- 验证正确性
- 快速暴露线程同步和数据结构问题

建议算例：

- `Tet4 6x6x6` 立方体分解
- `Tet4 10x10x10` 立方体分解

#### 第二阶段：规模扩展验证

目标：

- 观察并行开销与规模效应
- 初步比较图着色、直接累加、缓冲归并的趋势

建议算例：

- `Tet4 20x20x20`
- `Tet4 40x40x40`
- `Tet4 60x60x60`

其中 `60x60x60` 分解后的 `Tet4` 单元规模已经接近百万级，可作为真实工程案例前的桥接测试。

#### 第三阶段：真实工程验证

目标：

- 在 `3d-WindTurbineHub.inp` 上验证算法是否仍可运行且结论是否成立
- 比较规则网格与真实非结构化网格的差异

要求：

- 优先支持 `C3D4`
- 输入解析耗时单独统计
- 与规则网格结果对照分析

### 9.2 线程配置要求

每个算例至少测试以下线程数：

- 1 线程
- 2 线程
- 4 线程
- 8 线程
- 16 线程

对于真实工程案例，应优先覆盖“开发机物理核心数附近”的配置，以观察最有价值的实际性能。

### 9.2A 平台测试顺序要求

实验执行顺序必须采用：

1. `macOS + Mac Studio` 本地先完成功能正确性与早期性能验证；
2. 在需求、构建和脚本稳定后，再迁移到 `Windows + Intel U7 265KF` 做交叉平台验证；
3. 平台迁移后，优先复现相同 benchmark 配置，再追加平台特有优化。

### 9.3 重复测试要求

每组测试建议至少采用：

- 2 次预热
- 5 次正式测量

最终报告应采用平均值，并保留标准差或波动范围。

### 9.4 公平比较要求

不同算法对比时必须保持：

- 同一编译器与编译选项
- 同一线程绑定策略
- 同一网格输入
- 同一 CSR 结构
- 同一局部刚度 kernel

否则结论不具备可比性。

## 10. 验收标准

### 10.1 必须完成

1. 形成统一 CPU benchmark 框架。
2. 支持结构化 3D `Tet4` 网格生成。
3. 支持 `3d-WindTurbineHub.inp` 中 `C3D4` 的读取与运行。
4. 完成串行基线、图着色、直接累加、线程私有缓冲归并至少四类实现中的三类并通过验证。
5. 输出可复现的 benchmark 数据表和结论。

### 10.2 视为项目成功的最低标准

满足以下全部条件即可认定项目达到“可行性验证完成”：

1. 在规则 `Tet4` 网格上至少三类算法运行成功并通过正确性验证。
2. 在 `3d-WindTurbineHub.inp` 上至少两类并行算法运行成功并通过正确性验证。
3. 至少一种并行算法在真实工程案例上对串行基线取得明确正加速。
4. 文档能够清晰说明不同算法的优势、劣势、适用场景与后续推荐方向。

### 10.3 优秀结果标准

若进一步满足以下条件，则可视为项目达到“研究结论较强”：

1. 至少一种非图着色算法在工程案例上优于图着色；
2. 规则网格与工程网格上的趋势能形成一致或可解释的对照结论；
3. 端到端时间与数值组装时间都能给出可信分析，而不是只报告局部最优数字。

## 11. 风险与应对

### 11.1 风险：真实工程网格预处理代价过高

表现：

- 邻接图构建过慢
- 图着色内存开销过高
- CSR 符号组装耗时过长

应对：

- 将“图着色预处理时间”与“数值组装时间”分开统计
- 优先构建更轻量的数据结构
- 在真实工程案例上允许跳过部分高代价算法，但必须说明原因

### 11.2 风险：直接累加算法性能不稳定

表现：

- 热点自由度导致原子冲突严重
- 线程数升高后加速比下降

应对：

- 增加不同线程数测试
- 对比结构化网格与真实网格差异
- 增加局部缓冲归并作为替代路线

### 11.3 风险：线程私有缓冲算法内存膨胀

表现：

- 大网格下局部 COO 或哈希表占用过高
- 归并阶段时间过长

应对：

- 控制缓冲粒度
- 采用分块归并
- 在需求上明确“额外内存也是评价指标”

### 11.4 风险：真实输入文件解析复杂

表现：

- `.inp` 中存在非核心 section
- 编号不连续或包含杂项定义

应对：

- 先支持最小必要子集
- 对不支持 section 做显式跳过
- 在日志中输出解析摘要，便于排错

## 14. 结论与执行建议

本项目不应再被定义为“把已有图着色代码改到 CPU 上继续试一试”，而应被定义为：

“基于统一实验框架，在 CPU 平台系统比较多类并行整体刚度矩阵组装算法，并在简单三维网格与真实 `C3D4` 工程网格上验证其正确性、性能与工程可行性。”

建议执行策略如下：

1. 主平台选 `cpu_parallel_stiffness_assembly`，不要继续以 `fem_assembly` 作为核心承载代码。
2. `fem_assembly` 的价值在于提供图着色 CPU 基线思想与已有实验经验。
3. 早期先用规则 `Tet4` 块体网格快速打通。
4. 中期必须把真实工程案例 `3d-WindTurbineHub.inp` 纳入验证。
5. 评价指标不能只看加速比，必须同时看预处理代价、内存占用、确定性与工程可维护性。
6. 从第一天起就按 `macOS Apple Silicon -> Windows Intel x86_64` 的迁移路径设计，避免后期再集中返工脚本、构建和平台抽象。

如果后续需要继续推进实现计划，下一步应在此需求文档基础上进一步拆出实施计划，包括代码模块改造顺序、每种算法的写入接口设计、benchmark 字段定义以及 `.inp` 解析器的实现边界。
